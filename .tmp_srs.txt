Bab 1. Pendahuluan


1.1 Tujuan

Dokumen ini adalah Software Requirements Specification (SRS) Final Consolidated Blueprint untuk FruityPOS. Seluruh keputusan desain, spesifikasi fungsional, arsitektur, skema database, dan artefak navigasi dari SRS v1.0 sampai v1.3 telah digabungkan menjadi satu blueprint tunggal yang otoritatif. Dokumen ini menggantikan semua versi SRS sebelumnya dan menjadi satu-satunya acuan untuk implementasi.

SRS ini disusun mengikuti standar IEEE 830-1998 dan berfungsi sebagai kontrak teknis antara pemangku kepentingan bisnis dan tim pengembang.


1.2 Ruang Lingkup

FruityPOS adalah sistem web terpusat untuk manajemen operasional ritel buah-buahan multi-toko dengan model Decentralized Store. Setiap toko adalah unit mandiri dengan gudang built-in, tanpa gudang pusat.

Dalam lingkup:

  11 modul: Data Master, Pengadaan &amp; Inbound, Pricing Engine, POS Offline, POS Online,

  Mutasi Stok, Waste Management, Stock Opname, Dashboard &amp; Laporan,

  Manajemen Pengguna, Manajemen Toko

  RBAC 4 role: Owner, Stockist, Kasir, Admin Online

  Offline-first POS dengan IndexedDB (Dexie.js)

  WAC untuk kalkulasi HPP per toko, hpp_baseline worst-case



Di luar lingkup:

  Akuntansi GL &amp; Payroll | Integrasi API marketplace langsung

  Integrasi hardware timbangan | Modul retur pelanggan (KD-09)

Bab 2. Deskripsi Umum Sistem


2.1 Perspektif Produk

FruityPOS menggunakan model Decentralized Store. Setiap toko adalah unit operasional mandiri: area gudang + area penjualan. Tidak ada gudang pusat. Semua pengadaan dan inbound langsung ke toko masing-masing. Supplier sama untuk semua toko, hanya beda alamat kirim (KD-06).

Owner memiliki visibilitas penuh ke seluruh toko melalui dashboard terpusat dan mengendalikan harga jual secara seragam melalui Pricing Engine. Harga jual ditentukan berdasarkan hpp_baseline = MAX(avg_cost) untuk menjamin margin minimum di semua lokasi (KD-04).

ARSITEKTUR MULTI-TOKO:



  [Owner] -- Dashboard Terpusat -- Pricing Engine (harga seragam)

     |

     +-- [Toko Serpong]  Stockist + Kasir + Admin  &lt;-- PO/Inbound sendiri

     +-- [Toko KG]       Stockist + Kasir + Admin  &lt;-- PO/Inbound sendiri

     +-- [Toko BSD]      Stockist + Kasir + Admin  &lt;-- PO/Inbound sendiri



  Mutasi stok: Toko &lt;-&gt; Toko (horizontal, peer-to-peer)

  Supplier: Tunggal, beda alamat kirim, ongkir terpisah dari HPP

2.4 Batasan &amp; Asumsi

Batasan: Tidak ada gudang pusat (KD-03). Harga seragam (KD-04). Supplier tunggal (KD-06). Ongkir terpisah HPP (KD-07). Diskon off-system (KD-08). Tidak ada retur (KD-09). Timbangan manual. Printer via browser. Mata uang: IDR.

Asumsi: Setiap toko punya gudang fisik. Supplier antar langsung. avg_cost beda karena timing reorder. Setiap lokasi punya browser modern. Internet tersedia sebagian besar waktu. Buah homogen (tidak per batch).

3.2 Service-Repository Pattern

Request -&gt; Controller (Form Request validation)

  -&gt; Service (business logic, WAC calc, notifications)

    -&gt; Repository (abstraksi data access)

      -&gt; Eloquent Model -&gt; MySQL



Response: MySQL -&gt; Model -&gt; Repository -&gt; Service -&gt; Controller

  -&gt; Inertia::render(&apos;Pages/Screen&apos;, [&apos;props&apos; =&gt; $data])

  -&gt; React component re-render

3.4 LocationScope Middleware

Auto-inject WHERE location_id = auth()-&gt;user()-&gt;location_id untuk Stockist, Kasir, Admin. Owner (location_id = NULL) bypass. Memastikan isolasi data per-toko 100%.

Bab 5. Keputusan Desain Arsitektural (10 KD)

Semua KD berstatus LOCKED. Tidak dapat diubah tanpa Change Request formal.
KD
Keputusan
Alasan

KD-01

WAC: avg_cost = ((stok_lama x cost_lama) + (stok_baru x hpp_baru)) / total

Buah homogen. 1 angka HPP/produk/toko.

KD-02

Harga TIDAK auto-adjust saat HPP berubah. Owner manual.

Sensitivitas harga pelanggan tinggi.

KD-03

Decentralized Store. Tidak ada gudang pusat.

Sesuai realita operasional.

KD-04

hpp_baseline = MAX(avg_cost) semua toko. Harga seragam.

Menjamin margin minimum di toko termahal.

KD-05

Setiap toko WAJIB 1 Stockist + 1 Kasir + 1 Admin.

Operasi lengkap per toko.

KD-06

Supplier tunggal, beda alamat kirim.

avg_cost beda karena timing reorder saja.

KD-07

Ongkir terpisah dari HPP. Biaya operasional.

Menjaga kesederhanaan kalkulasi HPP.

KD-08

Diskon: approval di luar sistem. Catat nilai + catatan wajib.

Bisnis kecil, approval formal terlalu heavy.

KD-09

Retur pelanggan: di luar scope.

POS Online hanya lokal. Risiko retur minimal.

KD-10

Reorder point: per produk/toko, real-time listener, cooldown 1 jam.

Buah fast-moving, cron harian terlambat.


5.1 Detail WAC

avg_cost = ((stok_lama x avg_cost_lama) + (stok_baru x hpp_mentah)) / total_stok



Contoh: Lama 200 Pcs x Rp 9.000 + Baru 200 Pcs x Rp 11.000 = Rp 10.000/Pcs



WAC BERUBAH: Inbound baru, Mutasi diterima di tujuan

WAC TIDAK BERUBAH: Penjualan, Waste, Opname (qty berubah, cost tetap)


5.2 Detail hpp_baseline

hpp_baseline = MAX(avg_cost) dari semua toko dengan stok &gt; 0

Owner set margin% dari hpp_baseline. Harga jual seragam.



Contoh: Serpong Rp 9.000, BSD Rp 11.000, KG Rp 9.500

hpp_baseline = Rp 11.000, margin 30% = selling_price Rp 14.300

Margin aktual: Serpong 37%, BSD 23%, KG 34%


5.3 Detail Reorder Point

TRIGGER: TransactionCompleted, WasteApproved, MutationShipped

  -&gt; DeductInventory -&gt; CheckReorderPoint



LOGIC: Jika qty &lt; min_quantity DAN last_notified_at &gt; 1 jam lalu:

  -&gt; Notif ke Stockist toko + alert Owner + update last_notified_at


5.4 Struktur Laba-Rugi

Revenue -- Diskon -- COGS (hpp_at_sale x qty) = Laba Kotor

Laba Kotor -- Ongkir -- Waste -- Mutasi Loss = Laba Operasional
